# 变更提案: multi-runtime-blog（单库 MVP：Workers + D1）

## 需求背景
- 现状：仓库中存在两套历史资产：
  - `oldblog/`：Astro + React 的老博客实现（含较完整的 UI 规范与页面/组件实现）。
  - `oldsql/`：Rust 的 SQL 构建/参数化/校验能力（可作为设计参考）。
- 目标：先重构出一个**可直接部署**的单库基础版本（Cloudflare Workers + D1），把核心链路跑通；视觉风格为“边缘化”（主文偏左 + 右侧页边栏 + 大留白）。未来再扩展多运行时/多数据库（本方案只预留扩展点，不实现）。

## 产品分析（触发：大规模重构 + 新架构）
### 目标用户与场景
- **读者**：快速浏览文章、按标签/时间筛选、搜索、阅读体验稳定（暗色模式、目录、阅读进度等）。
- **管理员（你）**：只需要一个管理员后台（无普通用户体系/无评论），完成内容写入、发布、资源上传、站点配置。
- **运维/部署**：同一套代码尽量覆盖多平台；允许多站点并行部署，用户手动选择站点。

### 核心痛点
- 部署形态割裂：不同平台运行时/数据库能力不同，容易导致实现发散；需要先有“可上线主线”。
- 数据来源不统一：希望**内容全部进入数据库**，后续页面渲染/搜索/RSS/Sitemap 都依赖 DB。
- UI 风格诉求明确但缺“可执行规范”：需要一套设计令牌与页面布局规则，保证实现一致性。

### 价值主张与成功指标
- **价值主张**：一套 TS 代码跑多个平台；统一 API 与适配层；前端与后端可“同源部署”；视觉呈现走一致的设计系统。
- **成功指标（可验证）**：
  - 功能齐全：文章列表/详情、筛选、搜索、RSS/Sitemap、暗色模式、关于/项目/书影等页面、管理端 CRUD。
- 单库可上线：Cloudflare Workers + D1 可跑通（读写/搜索/RSS/Sitemap/管理端/图片上传）。
- 扩展可预期：为未来多数据库/多运行时预留抽象边界（不影响 MVP 落地）。
  - “边缘化”风格落地：文章页在大屏下呈现“偏边 + 大留白 + 页边 TOC/信息”的阅读布局。

## 变更内容（单库 MVP 总览）
1. 后端：TypeScript + Hono（Workers 入口），先只支持单库 D1。
2. 内容：文章写入时渲染并清洗（Markdown → HTML → sanitize），DB 同时保存 `markdown` + `rendered_html` + `content_text`（用于 LIKE 搜索）。
3. 发布：支持草稿/已发布/定时发布（对外可见规则基于 `publish_at <= now`）。
4. 图片上传：优先使用 R2（可选绑定）；未绑定 R2 时禁用上传（仍可粘贴外链图片）。
5. 公开站 + 管理端：Web 以 Astro+React（后续实施时落地），Admin 以 Vite+React；对外同源 `/api/*`。
6. 设计系统：文章页采用 left 边缘化布局（主文偏左，右侧页边栏）。

## 影响范围
- **代码结构**：将从“旧代码并存”转为“新主线 + 旧资产参考/迁移脚本”。
- **数据层**：引入 migrations（至少能在 D1/SQLite 上跑通），并定义跨库 schema 约束。
- **前端**：以新设计系统重做 Layout/Header/文章页等；`oldblog/docs/ui-design/*` 作为复刻参考。
- **安全**：管理员登录、会话、上传资源、输入校验、权限隔离。

## 核心场景
### 需求: 阅读（列表/筛选/详情）
**模块:** web + api + db
- 场景: 读者访问文章目录
  - 期望：网格卡片 + 目录卡片；支持分页；暗色模式一致。
- 场景: 读者按标签/时间/排序筛选
  - 期望：筛选条件可组合；URL 可分享；结果可分页。
- 场景: 读者阅读文章详情
  - 期望：偏边布局（主文列偏左/偏右）；右侧页边 TOC；阅读进度；相关内容推荐；过期提示（可选）。

### 需求: 搜索
**模块:** web + api + db
- 场景: 读者在导航栏搜索
  - 期望：建议/补全（可选）；结果列表可无限加载（可选）；可键盘操作。

### 需求: 管理端（写入）
**模块:** admin + api + auth + db + storage
- 场景: 管理员登录后创建/编辑/发布文章
  - 期望：草稿/发布态；标签/分类；内容编辑（Markdown/富文本二选一，先 Markdown）。
- 场景: 上传图片/附件
  - 期望：存储可插拔（R2/S3/本地）；生成可公开访问的 URL。

### 需求: 多站点手动切换
**模块:** web + admin + api + config
- 场景: 用户选择站点并持久化偏好
  - 期望：站点列表来自静态 `site-config.json`；切换后不自动跨库同步；各站点 token 独立。

## 风险评估
- **Workers 资源限制**：需控制 CPU/内存/冷启动；渲染与清洗放在写入时，读取走预渲染 HTML。
- **搜索降级**：MVP 用 LIKE + 加权相关度，后续可引入 FTS（作为扩展点）。
- **上传与存储**：R2 未绑定时禁用上传；绑定后需做好鉴权、类型/大小限制、缓存头。
- **安全风险**：管理员入口暴露在公网；需避免 SQL 注入、CSRF、会话泄漏、上传滥用、暴力破解等。
