# 变更提案：编辑器“聪明实时预览”（方案 B）

## 需求背景
当前后台编辑器预览继续依赖服务端渲染接口 `POST /api/admin/render`（预览结果与线上一致），但前端在内容输入时采用固定 `350ms` 的 `setTimeout` 触发渲染，请求不可取消，导致：
- 快速连续输入时产生大量 in-flight 请求，即使 UI 通过 `previewSeq` 避免“旧结果覆盖新结果”，服务端仍会做多次无效渲染；
- 当服务端限流/变慢时，请求堆积更明显，易触发 `429`、WAF/Cloudflare 频控，以及 Worker CPU 压力升高。

参考计划文档：`docs/editor-preview-plan-b.md`。

## 目标
1. 继续使用 `POST /api/admin/render` 作为预览渲染的唯一来源（不引入本地 Markdown 渲染）。
2. 显著减少无效请求与并发堆积：保证同一时刻最多 1 个渲染请求 in-flight，并支持合并/取消。
3. 自动预览只在需要时触发：仅当 `layout=preview/split` 且页面可见时自动渲染。
4. 可恢复：遇到 `429/5xx` 自动退避、必要时暂停自动预览，但允许用户手动“刷新预览”。

## 非目标（本次不做）
- 不引入本地 Markdown 渲染（方案 A）。
- 不改动服务端渲染结果的语义（仅客户端触发策略与请求控制）。

## 影响范围
- **模块：** Admin 编辑器预览链路
- **文件：**
  - `apps/admin/src/api.ts`
  - `apps/admin/src/pages/EditorPage.tsx`

## 核心场景

### 需求：自动预览降低请求压力
**模块：** Admin 编辑器

#### 场景：快速连续输入
用户持续输入 10 秒：
- Network 面板中旧的 `/api/admin/render` 请求会被取消或被合并（不会无限堆积）；
- UI 不会出现“旧预览覆盖新预览”，并且预览更新频率会在慢渲染时自动下降。

#### 场景：切换到写作模式 / 切换浏览器 Tab
- `layout === "write"` 时不触发自动预览；
- `document.visibilityState !== "visible"` 时不触发自动预览；
- 从隐藏切回可见时可触发一次“落后补偿”的渲染（仅在需要预览时）。

#### 场景：触发限流或服务端错误
- `429/5xx` 时自动进入退避（例如 5s/10s/...），并提示“自动预览已降频/暂停，可手动刷新”；
- 连续失败达到阈值时自动关闭 `autoPreview`，由用户手动恢复。

## 验收标准（可验证）
1. 快速连续输入时，不再出现大量堆积的 `/api/admin/render` 请求；同一时刻最多 1 个请求 in-flight。
2. 在 `layout=write` 或页面不可见时不再后台刷预览。
3. 自动预览遇到 `429/5xx` 会自动退避/暂停，且手动“刷新预览”仍可强制渲染。
