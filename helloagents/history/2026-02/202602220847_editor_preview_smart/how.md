# 技术设计：编辑器“聪明实时预览”（方案 B）

## 技术方案

### 核心技术
- 前端：React（`apps/admin`）
- 后端：Worker 渲染接口 `POST /api/admin/render`（保持不变）

### 实现要点（客户端）
1. **请求可取消（AbortController）**
   - `renderAdminMarkdown(content_md, opts?)` 支持传入 `AbortSignal`；
   - 在 `EditorPage` 内维护 `AbortController` 引用，每次发起新渲染前取消上一次请求；
   - `AbortError` 视为正常取消，不提示错误、不影响 UI。

2. **触发条件收紧（gating）**
   - 自动预览仅在 `layout !== "write"` 时触发；
   - 自动预览仅在 `document.visibilityState === "visible"` 时触发；
   - 从隐藏切回可见时，如当前需要预览且内容已变化，触发一次补偿渲染（可走防抖）。

3. **防抖调整 + 自适应**
   - 默认防抖由 `350ms` 调整为 `800ms`；
   - 记录上一轮渲染请求耗时（从发起到响应/失败），若耗时偏高则提高下一次自动预览防抖（上限 `1500ms`）。

4. **429/5xx 退避与自动预览暂停**
   - 对 `429` 与 `5xx` 进入退避窗口：`5s/10s/...` 逐步增加并上限封顶；
   - 退避期间不触发自动预览（手动刷新仍可强制渲染）；
   - 连续失败达到阈值（例如 3 次）自动关闭 `autoPreview`，并提示用户可手动刷新或重新开启自动预览。

5. **请求合并（只保留“最新一份”）**
   - 同一时刻最多 1 个渲染请求 in-flight；
   - in-flight 期间若内容再次变化，仅记录“最新待渲染内容”；当前请求结束后再渲染最新内容（队列长度永远为 1）；
   - 保留 `previewSeq`（或等价序列）避免旧请求结果覆盖。

## API 设计
不新增接口；仅在 `apps/admin/src/api.ts` 中扩展客户端方法签名以支持 `AbortSignal`。

## 安全与性能
- 安全：不改变服务端鉴权/同源策略；客户端仅增加请求控制。
- 性能：通过“最多 1 个 in-flight + 退避 + 更长防抖 + 仅必要时触发”降低服务端渲染压力与触发限流的概率。

## 测试与验证
- 手动回归（关键场景）：
  - 快速连续输入 10 秒：Network 中请求应被取消/合并；
  - 切换到写作模式/切换浏览器 tab：不再后台刷预览；
  - 模拟 429/500：UI 提示合理，自动预览会降频/暂停且可恢复；
  - 手动“刷新预览”：即使自动预览关闭也能正常渲染。
